SB-LandOwner-Sync Project — Technical Specification

Document type: Technical Specification  |  Project: SB-LandOwner-Sync  |  Version: 1.0.0  |  Date: June 2026  |  Platform: MuleSoft Anypoint Platform 4.6.0  |  Proposed Design  |  Source: business-requirements.pdf

Overview

This integration synchronises land-rights owner data from Shikun & Binui's Rainbow system into Salesforce. The business need is to produce a "warning note" (הערת אזהרה), which requires an accurate, up-to-date list of who holds rights in a parcel of land and what share each holds.

Data originates from the Rainbow registrations, owners and businessPartners OData entities (database 262) and is written into the Salesforce custom object LandOwner__c. Per the business decision, records are not linked to a Salesforce Account and no Project is created — because Rainbow data is not always consistent and identifiers are sometimes missing, the information is written as a flat record directly into the "Land Owners" table via upsert. The integration runs as an unattended nightly batch.

4. Flows

4.1 Flow: landowner-sync-scheduler-flow

doc:id: sched-main-0001

TriggerExpressionDescription
Scheduler${sync.cron} — default 0 0 2 * * ? (02:00 daily, Asia/Jerusalem)Nightly batch that orchestrates the full Rainbow → Salesforce land-owner synchronisation

Processing Steps

1
Log start — emit START landowner-sync correlationId=#[correlationId]
2
Purge stale records — flow-ref to delete-stale-landowners-subflow (removes expired records, preserving manually-managed ones)
3
Fetch current registrations — flow-ref to fetch-registrations-subflow; result stored in vars.registrations
4
Iterate registrations — <foreach> over vars.registrations:
  • 4a. flow-ref to build-landowners-for-registration-subflow — fetch owners + business partner, build flat records
  • 4b. Accumulate built records into vars.landownerBatch
5
Filter empties — remove any registration that yielded zero owners (vars.landownerBatch filter (not isEmpty($)))
6
Upsert to Salesforce — flow-ref to upsert-landowners-subflow (batched upsert by external ID)
7
Log summary — emit END landowner-sync upserted=#[vars.upsertCount] deleted=#[vars.deleteCount]

4.2 Sub-flow: delete-stale-landowners-subflow

doc:id: sub-delete-0002   Sub-flow called by 4.1 step 2.

Processing Steps

1
Query expired records — Salesforce query: SELECT Id FROM LandOwner__c WHERE ToDate__c != null AND ToDate__c < TODAY AND Lawyer__c = null
2
Guard — <choice>: if no rows, skip delete and set vars.deleteCount = 0
3
Delete — Salesforce delete on the returned Id list; capture count into vars.deleteCount

Rule: a record is "stale" only when its ToDate__c has passed. Records with an empty ToDate__c are never deleted. Records where Lawyer__c was filled manually by Shikun & Binui are excluded from deletion so manual work is preserved.

4.3 Sub-flow: fetch-registrations-subflow

doc:id: sub-fetch-reg-0003   Sub-flow called by 4.1 step 3.

Processing Steps

1
Request registrations — HTTP GET on Rainbow_Analytics_HTTP_Request_config:
/registrations?$filter=isCurrentRegistration eq true
2
Page — follow OData @odata.nextLink until exhausted (collect all pages)
3
Map — project each registration to { registrationKey, buildingCode, propertyKey, projectCode, rightType, plot } into vars.registrations

4.4 Sub-flow: build-landowners-for-registration-subflow

doc:id: sub-build-0004   Sub-flow called by 4.1 step 4a, once per registration.

Processing Steps

1
Fetch owners — HTTP GET /owners?$filter=toDate eq null and registrationKey eq #[vars.reg.registrationKey]
2
Skip if no owners — <choice>: if owners list is empty, return [] (registration filtered out per business rule)
3
Enrich business partner — for each owner, HTTP GET on Rainbow_Operation_HTTP_Request_config: /businessPartners?$filter=id eq #[owner.businessPartnerId] to obtain idtype, name, address, legal id
4
Build flat records — DataWeave transform combining registration + owner + business partner into LandOwner__c shape (see Section 9.1)
5
Resolve Building lookup — <choice>: if buildingCode present, set Building__r.BuildingCode__c for the relationship; otherwise leave building null (record linked at project level only)

4.5 Sub-flow: upsert-landowners-subflow

doc:id: sub-upsert-0005   Sub-flow called by 4.1 step 6.

Processing Steps

1
Chunk — split vars.landownerBatch into chunks of ${salesforce.batchSize} (default 200)
2
Upsert — Salesforce upsert on LandOwner__c using external ID field External_Key__c
3
Collect results — accumulate success/failure counts; log any item-level failures with their External_Key__c

Response / Outcome

OutcomeChannelDetail
SuccessLogEND landowner-sync upserted=N deleted=M
Partial failureLog (WARN)Per-record External_Key__c + Salesforce error message
Fatal failureGlobal error handlerLogged at ERROR; run marked failed (see Section 6)

6. Error Handling Strategy

Error TypeScopeBehaviour
HTTP:CONNECTIVITY / HTTP:TIMEOUT (Rainbow)On Error Continue (per registration)Retry up to 3× with backoff; on final failure, log the failing registrationKey and continue with the next registration (no abort)
SALESFORCE:CONNECTIVITYOn Error Propagate (global)Abort run, log at ERROR; next nightly run retries the full set (idempotent upsert)
SALESFORCE:INVALID_FIELD / item failuresOn Error Continue (upsert)Log per-record error with External_Key__c; remaining records still committed
ANY (uncaught)Global default error handlerLog full error + correlationId at ERROR level

9. Data Transformation Detail

9.1 Build flat LandOwner__c record — combines a Rainbow registration, its owner, and the owner's business partner into a single flat Salesforce record. Field mapping is taken directly from the requirements document. External_Key__c is the Rainbow owner record id (confirmed), used for idempotent upsert. Project__c is written as an external-id lookup relationship (Project__r).

DataWeave: 
%dw 2.0
output application/java
// inputs: vars.reg (registration), owner, bp (businessPartner)
---
{
  External_Key__c: owner.id as String,            // Rainbow owner record id (upsert key)
  Name: (bp.buisnesspartnername default "")[0 to 79],   // required, max 80
  Idtype__c: bp.idtype,                                // ח.פ / ת.ז
  IdNumber__c: owner.buisnesspartnerlegalidnumber default bp.buisnesspartnerlegalidnumber,
  Address__c: bp.buisnesspartneradress,
  Essence__c: vars.reg.righttype,                      // מהות הזכות
  OwnershipPercent__c: owner.ownerspercent,
  // Project__c is a lookup — related by external code via Project__r
  Project__r: { Project_Code__c: vars.reg.project_code },   // mandatory
  BuildingCode__c: vars.reg.buildingCode,
  Plot__c: vars.reg.plot,
  FromDate__c: owner.fromdate as Date,
  ToDate__c: owner.toDate as Date default null
}
9.2 Conditional Building lookup — when a buildingCode exists the record is related to a Building__c via external-id relationship; otherwise the building reference is omitted and the record stands at project level only.

DataWeave: 
%dw 2.0
output application/java
---
payload map ((rec) ->
  if (rec.BuildingCode__c != null)
    rec ++ { Building__r: { BuildingCode__c: rec.BuildingCode__c } }
  else
    rec - "BuildingCode__c"
)

12. Design Assumptions & Open Questions

The four high-impact items below were confirmed by the client and are reflected in the design above. The remaining open questions are operational and do not block development.

Confirmed Decisions

ItemConfirmed decision
Stale-record purge driverAdd ToDate__c on LandOwner__c mapped from owners.toDate; delete where ToDate__c < TODAY AND Lawyer__c = null (empty ToDate__c and lawyer-managed records preserved) Confirmed
Upsert external keyExternal_Key__c = Rainbow owner record id (owner.id) Confirmed
Rainbow OData authenticationHTTP Basic — service username / password (secure properties) Confirmed
Project__c populationLookup relationship written by external code via Project__r.Project_Code__c = project_code Confirmed

Remaining Design Assumptions

AssumptionRationaleImpact if wrong
Mule runtime 4.6.0, Java 17, batch size 200, 3 retriesCurrent supported defaults; no NFRs givenMay need tuning for very high registration volumes
Building is matched by BuildingCode__c as an external-id relationship on Building__cSpec says "link to building via lookup if building exists" but not the match keyIf Building lookup uses a different key, the relationship insert fails
Schedule fixed at 02:00 Asia/JerusalemSpec says "once nightly" without a timeMay clash with Rainbow maintenance windows
